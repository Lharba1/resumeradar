Synthesis

Most Likely Failure

No auth kills retention before the product can grow. Every user who uploads a CV and generates a tailored CV loses all of it when they close the tab. They return to a blank slate. The app acquires users continuously but retains none — a leaky bucket. Every growth lever (alerts, saved searches, referrals) is structurally unavailable.

Most Dangerous Failure

One viral post triggers a $42,000/month API bill overnight. 2,000 daily users × 42 GPT-4o calls = $420/day. One bot operator discovers the open endpoint: $1,200/day. OpenAI suspends the account mid-month. The app goes dark with users mid-workflow, no warning, no graceful shutdown.

The Hidden Assumption

"Users complete their full job-search workflow in one session, and the product delivers its entire value in that session."

This was never a stated decision — it was never decided at all. The entire architecture defaulted to it, and it cascades into: no auth, no session persistence, no alerts, no retention, no monetization path.

Failure Deep-Dives

Job Bank scraper breaks or gets blocked
High Severity
The Failure Story

ESDC pushes a routine redesign — React-rendered layout replaces JSP-era HTML. The article.action-buttons wrapper gets renamed. Cheerio returns zero matches silently. The API's catch block swallows failed queries; the allRaw.length === 0 branch returns a 404 indistinguishable from "no jobs for this title." Users assume they're unqualified and tweak search terms. No alert fires because HTTP 200s are still being served.

By month 3 the builder notices but Job Bank has also added bot detection — a Cloudflare challenge page that requires JavaScript. The static fetch() hits the challenge wall and returns a 403, which gets caught and silenced inside Promise.allSettled.

By month 6 the site requires JS execution for search results. Cheerio cannot run JS. The entire job-feed feature is dead until rebuilt from scratch with Playwright or a paid scraping API.

The Underlying Assumption
jobbank.gc.ca is a stable, scrapable, static HTML site that will not add bot protection, change its DOM, or require JS rendering — indefinitely, without notice.
Early Warning Signs
  • Result counts quietly drop (15 → 3 → 0) with no server errors
  • article.action-buttons returns 0 matches on a manual curl test while the browser still shows jobs
  • HTTP 200 responses with a Cloudflare challenge body (cf-browser-verification, __cf_bm cookie)
  • jsessionid disappears from Job Bank URLs (signals a backend migration)
OpenAI API costs spiral out of control
Critical Severity
The Failure Story

A developer posts ResumeRadar on Reddit. Traffic spikes to 200 daily users. Each session triggers ~42 GPT-4o calls (20 visa classifications + 20 scoring + 1 CV + 1 insights). At ~$0.005/call: $42/day — manageable. A Hacker News mention brings 2,000 daily users for a week: $420/day.

A bot operator discovers the unprotected endpoint and scripts 500 automated sessions/hour. 20 job descriptions × 500 sessions = ~10M tokens/hour. At $5/1M input tokens: $50/hour, $1,200/day from bots alone. Real users add $200/day. Monthly bill: ~$42,000. OpenAI suspends the account. The app goes dark.

The Underlying Assumption
Usage will be low-volume and human-paced. Organic growth will scale slowly enough to monetize before costs become serious. The open endpoint is a convenience, not a liability.
Early Warning Signs
  • Daily OpenAI spend exceeds $20 within the first two weeks of any public announcement
  • Requests cluster in sub-second intervals in the usage dashboard (bot signature)
  • A single IP accounts for >10% of total monthly token consumption
  • App latency increases as parallel calls multiply — feels slow before the bill arrives
No authentication — zero retention
Critical Severity
The Failure Story

A user uploads their CV, spends 20 minutes tuning their profile, scores three promising leads, generates a tailored CV, and closes the tab. The next morning: blank slate. UUID is lost, sessionStorage is gone. They have no reason to believe the app knows them. They Google a competitor with accounts and never return.

Word of mouth never compounds. Power users — the ones who would revisit weekly and evangelize — churn at the exact moment they'd have become loyal. Growth looks like a leaky bucket: high daily uniques that never convert to weekly actives. The app also can't send job alerts, follow-up nudges, or re-engagement emails.

The Underlying Assumption
The core value is delivered in a single session. Retention was treated as a product problem to solve later, after the session-only architecture was already baked in.
Early Warning Signs
  • Near-zero repeat sessions from the same device in analytics
  • Zero direct/bookmark traffic growth despite new-user spikes
  • User feedback requests "how do I save my profile"
  • Reviews mentioning data loss
Promises four markets, delivers one
High Severity
The Failure Story

A user in Morocco sees "Canada, USA, New Zealand, and Australia" on the landing page. They select "Australia," search their title, and get zero results — or worse, Canadian jobs tagged as Australian in the database. They conclude the product is broken, or that Australian employers aren't hiring. Both conclusions are wrong, and both erode trust permanently.

There is no graceful degradation — no "coming soon" label, no disabled state, no honest message. The UI actively invites the broken path. The error message even leaks "Job Bank" regardless of country selected: a visible tell that the backend is country-agnostic.

The Underlying Assumption
"Country" is a display attribute — a label to attach to data — not a routing key that determines which data source to call. The geographic scope was designed at the UI layer and deferred indefinitely at the data layer.
Early Warning Signs
  • The country field is written to the database but never read back as a filter or router
  • QUERY_GENERATOR_SYSTEM prompt hardcodes "Canadian job boards" while the UI advertises four markets
  • The error message leaks "Job Bank" for any country selection
CV quality insufficient for immigrant stakes
High Severity
The Failure Story

Amara, a software engineer from Senegal with 8 years of experience, arrives in Montreal and uploads her profile to ResumeRadar. She generates a CV and submits it to 40 employers over two weeks. She hears nothing. What she doesn't know: GPT-4o quietly dropped her most recent role, reworded accomplishments into vague generalities, and invented a soft-skills bullet she never claimed.

She has no baseline to compare against. She doesn't know what a strong Canadian CV looks like. She iterates on the wrong variables — her cover letter, her LinkedIn — while the CV quietly poisons every application. Weeks become months. Her work permit has a clock. Each failed cycle costs confidence, money, and legal runway. The product presents as authoritative ("ATS-optimized") while producing output no expert ever reviewed.

The Underlying Assumption
GPT-4o, given structured input, will produce output that is accurate, complete, and locally appropriate — without verification. The user will catch any errors.
Early Warning Signs
  • Users regenerate CVs multiple times without understanding why (silent dissatisfaction)
  • No user ever reports a specific factual error — they can't detect what they can't compare
  • Download rates are high but callback rates (if tracked) are near zero
  • Generated CV omits roles or conflates dates — detectable with automated output diffing
Legal exposure: scraping ToS + unprotected PII
High Severity
The Failure Story

Job Bank's legal team notices unusual automated traffic and sends a cease-and-desist citing their Terms of Use, which prohibit automated harvesting. Simultaneously, a user discovers their full name, address, and phone number in an exposed Supabase table. They file a complaint with the Office of the Privacy Commissioner under PIPEDA.

Two active legal fronts: scraping violation carries injunctive risk and contractual liability. The PIPEDA complaint triggers an OPC investigation — no consent form, no stated purpose, no retention limits, no deletion mechanism, no privacy policy. If any user is in the EU, GDPR exposure compounds: fines up to 4% of global revenue. Deleting the data now looks like evidence destruction. The app shuts down while legal counsel is retained.

The Underlying Assumption
"It's just a side project" creates a legal safe harbor. Collecting PII to help users is inherently benign, so compliance can be deferred until scale.
Early Warning Signs
  • Job Bank returns 403s or rate-limit blocks on scraped routes
  • A user asks "where is my data stored and can I delete it?"
  • Supabase dashboard shows anonymous public reads on the cv_profiles table
  • No /privacy URL exists and a user Googles it

5 — Medium Add /privacy page + Supabase RLS on cv_profiles Raw PII stored with no consent flow, no deletion mechanism, no access control. One misconfigured key from a PIPEDA complaint

Pre-Launch Checklist

  • Supabase Auth live — no AI endpoint callable without a valid session token
  • Rate limiter active — max 3 job searches + 2 CV generations per user per day (free tier)
  • Non-Canada country options show "Coming soon" — no silent broken paths
  • Privacy policy exists at /privacy — what data is stored and how to delete it
  • CV generation shows "Review carefully before submitting — AI can miss details from your profile"
